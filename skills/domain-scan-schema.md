# Schema Introspection

## When to use

Use `domain-scan schema` to discover what inputs a command accepts and what outputs it produces — at runtime, from the CLI itself. This replaces guessing flags or reading documentation. Always introspect before constructing `--json` payloads or `--fields` masks.

## Key commands

```bash
# List all available command names
domain-scan schema

# Get input/output schema for a specific command
domain-scan schema scan
domain-scan schema interfaces
domain-scan schema services
domain-scan schema methods
domain-scan schema schemas
domain-scan schema impls
domain-scan schema search
domain-scan schema stats
domain-scan schema validate
domain-scan schema match
domain-scan schema prompt

# Get all schemas in one call
domain-scan schema --all
```

## Rules

- **Always use `domain-scan schema <command>` before constructing a `--json` payload.** The schema tells you the exact field names, types, required/optional status, and defaults. Never guess.
- **Always use `domain-scan schema <command>` before constructing a `--fields` mask.** The output schema tells you which fields exist on the response type. Invalid field names produce a structured error.
- Use `schema --all` when you need the full API surface in one call (e.g., to build a tool integration or generate client code).
- Schema output is generated from Rust types at compile time via `schemars` — it is always in sync with the actual CLI behavior.

## Schema output structure

Each command schema contains:

```json
{
  "command": "interfaces",
  "description": "List interfaces/traits/protocols",
  "input": { /* JSON Schema for input/filter params */ },
  "output": { /* JSON Schema for output type */ }
}
```

- `input` describes the fields accepted by `--json` and the individual filter flags
- `output` describes the JSON structure returned by the command
- Both are standard JSON Schema (Draft 2020-12) generated by `schemars`

## Workflow: constructing a --json payload

1. Run `domain-scan schema interfaces` to get the input schema
2. Read the `input` object — note required fields, types, and defaults
3. Construct your JSON payload matching the schema: `domain-scan interfaces --json '{"name": ".*Repo", "show_methods": true}'`
4. If the payload is invalid, the error message references the schema and suggests the correct field names

## Workflow: constructing a --fields mask

1. Run `domain-scan schema interfaces` to get the output schema
2. Read the `output` object — note the top-level and nested field names
3. Construct your field mask: `domain-scan interfaces --output json --fields name,methods,file`
4. For nested fields, use dot notation: `--fields methods.name,methods.is_async`

## Common mistakes

- Guessing flag names or JSON field names without checking the schema → the CLI uses exact Rust field names (snake_case). `showMethods` won't work; `show_methods` will. Check the schema.
- Using `schema --all` when you only need one command → returns a large payload. Use `schema <command>` for a single command.
- Assuming the schema is static documentation → it's generated from compiled Rust types. If the CLI is updated, the schema updates automatically. Always re-introspect after upgrades.
- Ignoring the `input` schema when using individual flags → the schema shows which flags exist and their types, even if you're not using `--json`.
